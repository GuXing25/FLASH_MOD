# FLASH_MODLE 总路线

## 1. 项目最终目标

本项目的目标不是继续为每一颗 Flash 芯片手写一套独立 C++ 模型，而是形成一个具有统一性的 Flash 模型框架。

外部 AI 从不同数据手册中提取必要参数和行为描述后，输出结构化配置文件；FLASH_MODLE 读取该配置，通过统一内核、能力模块和器件策略组合出对应的 Flash 命令级仿真模型。

目标链路如下：

```text
数据手册
  |
  v
外部 AI 提取参数、命令、寄存器、约束和证据
  |
  v
结构化配置文件
  |
  v
FLASH_MODLE
  |
  +-- Common Core: NAND/NOR 共有物理与命令行为
  +-- Capability Modules: 可选能力模块
  +-- Device Policy: 少量厂商或器件族私有规则
  |
  v
对应器件的 Flash 行为模型
```

## 2. 当前阶段判断

已有工作已经完成第一阶段：从多类 NOR Flash 和 SPI-NAND Flash 中抽象出公共模型框架，覆盖了共同数据路径和共同约束，例如：

- NOR direct array read / page program / erase。
- SPI-NAND page read -> cache -> read from cache。
- SPI-NAND program load -> program execute。
- 擦除态为 `0xFF`。
- program 采用 `old & new`，只允许 bit 从 `1` 写成 `0`。
- WEL/WIP 或 OIP/BUSY 状态。
- 基本时序推进。
- 基本 Feature/Register 默认值。
- 部分 NAND ECC reserved spare 区、partial program、顺序编程约束。

当前不足也很明确：统一内核只覆盖公共行为，一些厂商私有命令、特殊寄存器位、复杂保护规则、OTP/Security Register、Copy-back、Read Retry、Die/Plane 选择等还没有系统性进入框架。

因此下一阶段的重点不是继续把所有特殊行为直接塞进 `flash_core.cpp`，而是先建立稳定的分类边界。

## 3. 核心架构原则

最合理的方向是：

```text
统一内核 + 可配置能力模块 + 器件策略层
```

不要走两个极端：

- 不继续把每颗芯片写成独立模型，否则无法形成统一框架。
- 不试图用一个纯参数配置解释所有 datasheet 行为，否则配置会变成复杂脚本语言，难以验证。

推荐比例：

```text
80% 通用行为：配置化
15% 可选能力：模块化
5% 厂商特例：policy 化
```

## 4. 三类差异处理方式

### 4.1 纯参数差异

这类差异适合配置文件表达。

典型字段：

- 器件类型：NOR / SPI_NAND / RAW_NAND。
- 容量、页大小、sector/block 大小。
- main/spare 大小、pages per block、blocks、planes。
- JEDEC ID、Manufacturer ID、Device ID。
- 典型/最大时序。
- 默认状态寄存器、Feature Register。
- 是否需要 WEL、是否默认全保护、是否要求 QE。

这部分应尽量保持声明式，不需要 C++ 扩展。

### 4.2 可选能力差异

这类差异不是单纯参数，但又不是某一家厂商独有，适合做成 capability module。

候选能力模块：

- `quad_enable`
- `four_byte_address`
- `block_protect`
- `suspend_resume`
- `otp`
- `security_register`
- `ecc_status`
- `bad_block_management`
- `copy_back`
- `die_select`
- `plane_select`
- `read_retry`

能力模块负责：

- 声明该能力需要哪些配置字段。
- 在模型初始化时挂载对应状态。
- 在命令执行前后参与校验或状态更新。
- 提供该能力自己的 behavior test。

### 4.3 厂商私有规则差异

这类差异不适合无限参数化，应放到 Device Policy。

例如：

- 某厂商 Feature Register 位含义特殊。
- 某些保护位组合含义复杂。
- 某颗芯片对 Copy-back、OTP lock、Security Register 有特殊限制。
- Read Retry 流程依赖私有寄存器序列。
- 独立 block lock、SPB/DPB、WPS 等组合规则。

Policy 不应该变成另一套单芯片模型。它只处理最后少量无法用公共模块表达的规则。

推荐层次：

```text
GenericNorPolicy
GenericSpiNandPolicy
  |
  +-- VendorFamilyPolicy
        |
        +-- 少量 chip override
```

## 6. 配置文件升级方向

当前平铺 `.conf` 适合第一阶段验证。最终建议升级为分层 schema。

建议字段组：

- `device`
- `geometry`
- `timing`
- `commands`
- `registers`
- `capabilities`
- `constraints`
- `policy`
- `source`
- `unsupported_features`

外部 AI 的职责不是生成 C++，而是生成符合 schema 的配置文件，并给出来源证据。

示例：

```yaml
schema_version: 1

device:
  name: W25N01GV
  class: SPI_NAND
  manufacturer: Winbond
  id: [0xEF, 0xAA, 0x21]

geometry:
  main_size: 2048
  spare_size: 64
  pages_per_block: 64
  blocks: 1024

timing:
  page_read_us: 60
  program_us: 250
  block_erase_us: 2000

commands:
  read_id: true
  page_read: true
  read_from_cache: true
  program_load: true
  program_execute: true
  block_erase: true
  reset: true

capabilities:
  ecc_status: true
  block_protect: true
  bad_block_management: true
  copy_back: false

registers:
  feature_a0_default: 0x7C
  feature_b0_default: 0x18
  feature_c0_dynamic: true

constraints:
  require_wel: true
  default_all_protected: true
  strict_sequential_program: true
  max_partial_programs: 4
  ecc_reserved_start: 2048
  ecc_reserved_end: 2112

policy:
  name: generic_spinand

source:
  datasheet: W25N01GV_datasheet.pdf
  page: 34
  confidence: 0.92

unsupported_features:
  - read_retry
```

## 7. 配置验证器

模型运行前必须先验证配置，而不是等执行命令时才发现结构错误。

基础规则：

- `class=NOR` 必须有 `memory_size/page_size/sector_size`。
- `class=SPI_NAND` 必须有 `main_size/spare_size/pages_per_block/blocks`。
- ECC reserved 区间不能超过有效 page size。
- NOR-only 命令不能和 SPI-NAND-only 几何混用。
- `copy_back`、`bad_block_management` 这类能力默认只允许 NAND-like 器件启用。
- `four_byte_address` 默认只允许 NOR-like 器件启用。
- policy 必须能被框架识别，不能静默忽略。

验证器输出应包含：

- error：阻止模型生成。
- warning：允许生成，但提示配置可能不完整。
- evidence gap：提醒哪些字段缺少数据手册来源。

## 8. 测试路线

每个配置生成模型后，都要跑统一行为测试。

公共测试：

- read-id。
- write-enable / write-disable。
- busy / wait-ready。
- erase -> read all `0xFF`。
- program -> readback。
- 二次 program 的 `old & new`。

NOR 测试：

- sector erase。
- block erase。
- chip erase。
- page program wrap。
- QE / Quad read-program gating。
- 4-byte address。
- suspend/resume。
- block protect / security register。

SPI-NAND 测试：

- page read -> cache read。
- program load -> program execute。
- block erase。
- Feature A0/B0/C0/D0。
- ECC reserved spare 区限制。
- partial program count。
- strict sequential program。
- bad block marker。
- copy-back。

## 9. 推进顺序

### 阶段 1：冻结当前公共能力边界

整理当前统一框架支持和不支持的能力，形成能力边界表。

### 阶段 2：建立正式 schema

确定外部 AI 必须输出的配置字段、字段含义、默认值、合法范围和来源证据格式。

### 阶段 3：实现 Config Loader / Validator / Builder

让框架可以从分层配置创建 `FlashModel`，并在创建前完成合法性检查。

### 阶段 4：拆出 Capability Module

优先拆通用度高的能力：

1. `quad_enable`
2. `block_protect`
3. `ecc_status`
4. `four_byte_address`
5. `security_register`
6. `otp`
7. `bad_block_management`
8. `copy_back`

### 阶段 5：引入 Policy

先保留：

- `generic_nor`
- `generic_spinand`

再根据已有模型沉淀：

- `winbond_family`
- `micron_family`
- `gigadevice_family`

### 阶段 6：建立 golden behavior test

每个配置至少有一组统一自测；每个能力模块有独立 behavior test。

## 10. 当前新目录的实现定位

`FLASH_MODLE` 作为下一代架构实验目录，第一版先实现：

- 分层配置样例。
- 配置加载器。
- 配置验证器。
- 模型构建器。
- 公共 NOR / SPI-NAND 最小命令路径。
- capability module 注册与校验骨架。
- generic policy 注册与选择骨架。
- demo NOR 和 demo SPI-NAND 自测。

它不是一次性替换旧 `FLASH_MOD`，而是先把最终架构形态搭起来。后续再逐步把旧目录里的成熟行为迁移进模块和 policy。

## 11. 第一轮迭代完成项

当前第一轮迭代已经形成一个可运行闭环：

- `make run-nor`：运行 NOR demo 自测。
- `make run-nand`：运行 SPI-NAND demo 自测。
- `make test`：运行 smoke test。
- `--validate-only`：只加载配置并执行 schema/语义验证。
- `configs/invalid_nor_with_nand_command.yaml`：负例配置，用来确认 validator 能阻止 NOR/NAND 命令混用。

当前已经进入行为自测的 capability/constraint：

- `block_protect`
- `ecc_status`
- `strict_sequential_program`

第二轮迭代补充：

- NOR `block_protect` 已支持通用保护窗口：`nor_protect_start/nor_protect_length`。
- `make test` 已覆盖 NOR 保护窗口内 program/erase 失败、窗口外正常读写擦。

第三轮迭代补充：

- 新增 `AddressMapper`，将 NOR sector 对齐、保护范围重叠、NAND page/block offset 从 `FlashModel` 中拆出。
- 新增 `tests/address_mapper_test.cpp`，`make test` 会先验证地址映射公式，再跑端到端 smoke test。

第四轮迭代补充：

- 新增 `RegisterEngine`，将 WEL、SR1/SR2/SR3、Feature Register、P_FAIL/E_FAIL 状态从 `FlashModel` 中拆出。
- 新增 `tests/register_engine_test.cpp`，`make test` 覆盖 WEL 位同步、Feature C0h 动态状态、P_FAIL/E_FAIL 合成。
- 为 `FlashModel`、`AddressMapper`、`RegisterEngine`、`CapabilityModule`、`StorageBackend` 补充职责边界注释。

第五轮迭代补充：

- 新增 `TimingEngine`，将 now_us、busy、busy_until、wait-ready 和普通时间推进从 `FlashModel` 中拆出。
- 新增 `tests/timing_engine_test.cpp`，`make test` 覆盖 advance、busy 自动完成、wait-ready、clear-busy。
- `FlashModel` 现在通过 `TimingEngine` 启动 program/erase/page-read busy，并通过它生成 OperationResult 的 now_us。

下一轮建议优先迁移：

1. NOR status register protect / BP-TB-CMP 位到保护窗口的 policy 解码。
2. NOR `four_byte_address`。
3. NOR `security_register`。
4. SPI-NAND `bad_block_management`。
5. SPI-NAND `copy_back`。
