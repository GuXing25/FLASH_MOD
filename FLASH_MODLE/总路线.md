# FLASH_MODLE 总路线

## 1. 项目目标

本项目的目标不是继续为每一颗 Flash 芯片手写独立模型，而是形成一个具有统一性的 Flash 模型框架。

外部 AI 负责阅读数据手册，提取必要参数、命令、寄存器、时序、约束和来源证据，并输出结构化配置文件。本框架负责加载配置、验证配置、选择能力模块和器件策略，最终生成对应器件的 Flash 命令级行为模型。

```text
External AI
  |
  v
Structured Config + Source Evidence
  |
  v
Config Validator
  |
  v
Flash Model Builder
  |
  +-- Common Core
  |     +-- storage
  |     +-- address mapping
  |     +-- timing engine
  |     +-- command transaction
  |     +-- interface timing
  |     +-- register/status base
  |     +-- 1 -> 0 program semantics
  |
  +-- Capability Modules
  |     +-- otp
  |     +-- security_register
  |     +-- block_protect
  |     +-- four_byte_address
  |     +-- suspend_resume
  |     +-- ecc_status
  |     +-- bad_block
  |     +-- copy_back
  |     +-- unique_id
  |     +-- sfdp
  |     +-- parameter_page
  |
  +-- Device Policy
        +-- generic_nor
        +-- generic_spinand
        +-- winbond_family
        +-- micron_family
        +-- gigadevice_family
        +-- boya_family
        +-- macronix_family
```

## 2. 核心判断

最好的路线不是“每颗芯片一个独立 C++ 模型”，也不是“一个超复杂配置解释器覆盖所有私有行为”。

更稳的结构是：

```text
统一内核 + 可配置能力模块 + 器件策略层
```

推荐取舍是：

```text
80% 通用行为：配置化
15% 可选能力：模块化
5% 厂商特例：policy 化
```

这样既能让外部 AI 只输出结构化配置，又不会让配置文件承担过于复杂、难验证的行为解释任务。

## 3. 差异分类

### 3.1 纯参数差异

适合放入配置文件。

- 容量。
- page/block/sector 大小。
- 时序。
- JEDEC ID。
- 默认寄存器值。
- OTP 页数。
- SFDP 空间大小。
- security register 数量和大小。

### 3.2 可选能力差异

适合做成 capability module。

- OTP。
- Security Register。
- Block Protect。
- 4-byte Address。
- Suspend/Resume。
- ECC Status。
- Bad Block Management。
- Copy-back。
- Unique ID。
- SFDP。
- Parameter Page。
- Die/Plane Select。
- Read Retry。

### 3.3 厂商私有规则差异

适合做成 device policy。

- 厂商保护位组合含义。
- 特殊 lock 规则。
- 特殊 read retry 表。
- 特定 feature register 行为。
- 上电默认状态的厂商族差异。

## 4. 配置文件方向

配置文件必须分层，不能长期保持平铺参数。

推荐结构：

```yaml
schema_version: 1

device:
  name: W25N01GV
  class: SPI_NAND
  manufacturer: Winbond

geometry:
  main_size: 2048
  spare_size: 64
  pages_per_block: 64
  blocks: 1024

timing:
  page_read_us: 60
  program_us: 250
  block_erase_us: 2000

commands:
  read_id: true
  page_read: true
  read_from_cache: true
  program_load: true
  program_execute: true
  block_erase: true
  reset: true

capabilities:
  ecc_status: true
  block_protect: true
  otp: true
  bad_block_management: true
  copy_back: false

registers:
  feature_a0_default: 0x7C
  feature_b0_default: 0x18
  feature_c0_dynamic: true

constraints:
  ecc_reserved_start: 2048
  ecc_reserved_end: 2112
  strict_sequential_program: true

policy:
  name: generic_spinand

source:
  datasheet: W25N01GV.pdf
  page: 34
  confidence: 0.92
```

外部 AI 的核心任务不是生成 C++，而是生成符合 schema 的配置，并尽量带字段级来源证据。

## 5. 当前框架状态

当前代码已经具备以下骨架：

- `include/flash_model`：公共接口。
- `src/config`：配置结构、加载和验证。
- `src/core`：地址、存储、时序、寄存器、命令事务、接口耗时。
- `src/capabilities`：可选能力模块。
- `src/policies`：通用策略和厂商族策略落点。
- `src/model`：命令级模型外观层。
- `src/builder`：配置到模型的组装入口。
- `src/cli`：demo 和验证入口。
- `tests`：单元测试和 smoke test。

当前模型已经覆盖 NOR 和 SPI-NAND 的大量公共路径，并且已经开始支持 datasheet-derived profiles。

## 6. 后续优化顺序

### 阶段一：冻结公共模型边界

- 不把厂商规则塞进 `src/core`。
- 不把所有能力细节塞进 `src/model`。
- 不让配置文件变成难验证的复杂脚本。

### 阶段二：强化配置 schema

- 字段分层清楚。
- 每个关键字段可带 evidence。
- Validator 能发现 NOR/NAND 混用、能力和命令不一致、范围越界。
- schema 与 `ModelConfig`、loader、validator 保持同步。

### 阶段三：完善能力模块

优先能力：

- OTP lock。
- Security Register lock 细化。
- Quad Enable。
- Block Protect bit table。
- Read Retry table。
- Plane/Die 约束。
- ECC status bit decoding。

### 阶段四：完善器件策略层

优先策略：

- `winbond_family`。
- `micron_family`。
- `gigadevice_family`。
- `boya_family`。
- `macronix_family`。

每个策略只放“无法配置化、无法通用模块化”的最后一小部分规则。

### 阶段五：建立 profile 和 golden test

基础测试：

- read-id。
- erase/program/read。
- old & new。
- busy timing。
- status bit。
- feature register。
- OTP。
- ECC。
- block protect。
- copy-back。
- parameter page。

## 7. 当前原则

- 配置文件表达事实，不表达复杂程序。
- Capability Module 表达可复用能力。
- Device Policy 表达少量私有规则。
- Common Core 只做稳定基础机制。
- Builder 是配置到模型的唯一正式装配入口。
- README 和注释要帮助后续维护者快速判断代码落点。
