# 代码分层说明

`FLASH_MODLE` 的实现代码按“统一内核 + 可配置能力模块 + 器件策略层”的路线拆分。

## 目录职责

```text
src/config
  配置结构、YAML-like loader、schema/语义 validator。
  这是外部 AI 输出结构化 profile 后进入框架的第一站。

src/core
  Common Core，保存 NAND/NOR 共用的底层机制：
  address mapping、register/status、timing、storage backend、
  command transaction、chiplet interface timing。
  这一层不放厂商私有规则。

src/capabilities
  可选能力模块。
  每个模块通过 hook 参与命令执行前后的检查或状态更新。
  适合承载 OTP、ECC、bad block、copy-back 等多芯片共享能力。

src/policies
  器件策略层。
  用于 generic_nor、generic_spinand 和后续 vendor family policy。
  只承载无法配置化、无法通用模块化的少量私有规则。

src/model
  FlashModel 命令级外观层。
  负责串起 core、capability、policy，但不应承载大量私有行为。
  这一层可以编排命令流程，但不应变成厂商规则仓库。

src/builder
  从配置文件创建模型：
  load config -> create policy -> validate -> create capability modules -> build FlashModel。
  这是配置对象变成运行时模型的唯一正式入口。

src/cli
  demo/self-test 命令行入口。
  正式可执行文件名是 flash_model_sim。

tests
  单元测试和 smoke test。
  当前覆盖 address、storage、registers、timing、capability、command、interface、evidence、model feature。

include/flash_model
  公共接口头文件。
```

每个代码目录均有对应的 `README.md`，作为该层的局部说明入口。

## 命名约定

目录已经提供上下文，因此文件名保持短而直接：

- `address.hpp/cpp`，不使用 `address_mapper.hpp/cpp`。
- `timing.hpp/cpp`，不使用 `timing_engine.hpp/cpp`。
- `command.hpp/cpp`，表达命令 transaction 形态。
- `interface.hpp/cpp`，表达 chiplet interface timing。
- `capability.hpp/cpp`，不使用 `capability_module.hpp/cpp`。
- `policy.hpp/cpp`，不使用 `device_policy.hpp/cpp`。
- `model.hpp/cpp`，不使用 `flash_model.hpp/cpp`。
- `builder.hpp/cpp`，不使用 `model_builder.hpp/cpp`。

类型名可以比文件名更完整，用来保留语义边界，例如 `FlashModel`、`CapabilityModule`、`DevicePolicy`。

## 依赖方向

推荐依赖方向如下：

```text
cli
  -> builder
      -> config
      -> policies
      -> capabilities
      -> model
          -> core
          -> capabilities
          -> policies
```

约束：

- `core` 不反向依赖 `model`、`cli` 或具体 demo。
- `capabilities` 可以读取 `ModelConfig` 和 `RuntimeState`，但不要直接操作底层存储。
- `model` 可以调用 capability hook，但不要把 OTP、copy-back、read-retry 等私有行为长期写死在自身内部。
- `policies` 只处理少量无法配置化、模块化的厂商族规则，不回退成单芯片完整模型。
- `config` 负责结构化输入和合法性检查，不生成 C++ 行为代码。
- `builder` 负责组装，不执行模型命令。
